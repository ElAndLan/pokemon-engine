# Effect Types Catalog

Version: 0.5
Status: Final reusable-helper polish pass for Claude handoff
Project: Creature Game Maker

Purpose: define the reusable battle, item, ability, form, ruleset, and overworld effect model for a Pokemon-inspired creature-battler. This is not a move encyclopedia and it is not permission to implement everything early. It is the normalization contract that lets moves, items, abilities, forms, weather, terrain, hazards, rulesets, and field interactions compose from shared code.

This document feeds:

- `BATTLE_SYSTEM_SPEC.md`
- `DATA_SCHEMA.md`
- importer normalization
- move-effect validation
- AI battle simulation
- golden battle replay design

The existing project plan wins for timing. This catalog defines coverage and architecture.

---

## 0. Final Decision From This Polish Pass

The engine should have **few effect primitives** and **many reusable helpers**.

Most official-style mechanics should be one of these:

1. **Primitive operation** — small resolver operation implemented in Core.
2. **Condition definition** — data with hooks, duration, counters, source, scope, immunity/filter rules, and parameters.
3. **Preset/alias** — named move/item/ability/field mechanic composed from primitives and conditions.
4. **Ruleset policy** — generation/custom-format difference that changes calculations, eligibility, or hook ordering.
5. **Unsupported preserved metadata** — imported data that is stored for audit but not executable yet.

Do **not** create one function per move. Do **not** create functions named after official moves unless they are content presets generated by data.

Examples:

| Named Mechanic | Correct Representation |
|---|---|
| Burn | persistent condition with residual damage, physical damage modifier, immunity/cure policy. |
| Fire Spin / Wrap / Bind / Whirlpool | `deal_damage` plus `apply_condition(volatile:partial_trap)`. |
| Constrict-like speed drop | `deal_damage` plus `chance_gate` plus `modify_stat_stage(speed,-1)`. |
| Stealth Rock | side condition `entry_hazard_damage(type:rock, type_scaled:true)`. |
| Spikes | side condition `entry_hazard_damage(grounded_only:true, stack_count:1..3)`. |
| Sticky Web | side condition with `on_switch_in -> modify_stat_stage(speed,-1)`. |
| Rain Dance | field condition `weather:rain`. |
| Strength Sap | `heal_hp(formula:target_current_attack)` plus `modify_stat_stage(attack,-1)`. |
| Belly Drum | `condition_gate(enough_hp)` plus `pay_cost(hp:50%)` plus `set_or_reset_stat_stage(attack,max)`. |
| Salt Cure | volatile condition with typed residual table. |
| Future Sight / Doom Desire | slot condition with delayed damage action. |
| Healing Wish / Lunar Dance | `pay_cost(faint_self)` plus slot condition that heals next replacement. |
| Mega / Gmax / Tera-like systems | `transform_form` plus ruleset eligibility and once-per-battle tracking. |

### Primitive Promotion Rule

A mechanic may become a new primitive only if it introduces a **new timing model**, **new state scope**, or **new resolver category** that cannot be expressed by:

- existing primitives
- condition hooks
- formula parameters
- move tags/filters
- ruleset policies
- preset composition

If it only changes numbers, targets, duration, source, filters, or hook behavior, it is not a new primitive.

## 1. Non-Negotiable Architecture Rules

1. **No per-move resolver classes.** A move is a sequence of data effects.
2. **Battle state owns truth.** UI submits actions and renders events; UI never mutates HP, status, stat stages, weather, inventory, party state, or progression directly.
3. **Effects resolve through deterministic hooks.** Hook ordering must be explicitly specified and golden-tested.
4. **Unsupported imported effects are disabled, not guessed.** The importer may preserve raw metadata, but runtime enablement requires every referenced primitive, hook, tag, condition, target shape, and ruleset policy to exist.
5. **Generation differences are data policies.** Do not fork the whole battle engine for Gen 3, Gen 4, Gen 9, Legends-style, or custom projects.
6. **One-off official mechanics become presets first.** Try preset composition, then condition hooks, then formula extension. New primitive is the final option.
7. **Every effect must emit events.** Damage, healing, status failure, immunity, duration expiration, item consumption, form change, and switch/capture outcomes must be observable through the event log.
8. **AI must use the same resolver.** AI score calculation may simulate, but it must not duplicate battle logic or bypass validation.
9. **Core must stay headless.** No UI, animation, sound, or renderer dependencies in effect logic.
10. **Effects must be serializable.** Every runtime effect state must support debug serialization and golden replay reproduction.

## 2. Effect Execution Model

### 2.1 Data Shape

A move/item/ability/event action should eventually normalize into this conceptual shape:

```json
{
  "id": "move:ember",
  "effects": [
    { "op": "deal_damage", "model": "standard" },
    {
      "op": "apply_condition",
      "condition": "status:burn",
      "target": "defender",
      "chance": 10,
      "filters": ["must_hit", "not_blocked_by_substitute_policy"]
    }
  ],
  "tags": ["protectable", "mirrorable"],
  "targeting": "selected_opponent",
  "rulesetProfile": "gen4_like"
}
```

The exact serialized schema belongs in `DATA_SCHEMA.md`. This document defines semantics.

### 2.2 Runtime Flow

```text
Player/AI chooses BattleAction
-> BattleController validates action
-> Resolver builds EffectContext
-> Hooks modify legality, target, type, priority, power, accuracy, damage, and secondary effects
-> Primitive operations resolve in deterministic order
-> Conditions/items/abilities/forms receive hook callbacks
-> BattleState mutates only through Core resolvers
-> BattleEvent log records every meaningful result
```

### 2.3 Required EffectContext Fields

`EffectContext` should be passed through almost every helper.

Required conceptual fields:

- `BattleState state`
- `RulesetProfile ruleset`
- `IRng rng`
- `BattleAction action`
- `CreatureInstance source`
- `CreatureInstance? target`
- `Side sourceSide`
- `Side targetSide`
- `MoveDef? move`
- `ItemDef? item`
- `AbilityDef? ability`
- `EffectSource sourceKind` (`move`, `item`, `ability`, `field`, `ruleset`, `event`, `debug`)
- `EffectId currentEffectId`
- `HookId currentHook`
- `TargetingResult targets`
- `DamagePacket? damagePacket`
- `List<BattleEvent> emittedEvents`
- `EffectTrace trace` for debug/golden replay

## 3. Canonical Primitive Operations

This is the locked primitive set for long-term coverage. Implementation should still be phased.

### 3.1 Gates, Chance, And Filters

| Primitive | What It Does | Covers |
|---|---|---|
| `eligibility_check` | Validates whether an action/effect can legally happen. | PP, no legal moves, Taunt, Assault Vest-like rules, trainer no-run/no-catch, target alive, capture legal. |
| `condition_gate` | Skips/fails an effect unless conditions are true. | first-turn-only, target asleep, weather active, enough HP, item held, grounded, species/form required. |
| `chance_gate` | Rolls whether an effect happens. | secondary status chance, stat-drop chance, flinch chance, item chance, thaw chance. |
| `effect_filter` | Selects/filter targets, moves, conditions, tags, source kinds, and scopes. | sound-only blocks, powder immunity, priority blockers, contact triggers, type-specific modifiers. |

### 3.2 Targeting, Action Order, And Turn Control

| Primitive | What It Does | Covers |
|---|---|---|
| `modify_targeting` | Changes legal target shape, redirect, random target, ally target, side target, or spread target. | Follow Me, Rage Powder, Spotlight, Ally Switch, random moves, spread moves. |
| `modify_priority` | Changes priority bracket for an action. | Quick Attack, Prankster-like effects, Gale Wings-like effects, terrain priority blockers. |
| `modify_turn_order` | Changes order after priority/speed rules. | Trick Room, After You, Quash, Custap/Quick Claw-like effects, Stall-like ordering. |
| `block_or_replace_action` | Prevents, delays, or replaces selected action. | sleep, freeze, paralysis, flinch, confusion self-hit, recharge, disobedience, Struggle fallback. |
| `lock_action` | Restricts future choices or forces repeats. | Encore, Torment, Disable, Imprison, Choice lock, Rollout/Outrage-style locked moves. |
| `call_or_copy_move` | Calls, copies, repeats, sketches, instructs, or selects a move from a pool/history. | Metronome, Copycat, Mirror Move, Sleep Talk, Assist, Instruct, Me First, Mimic, Sketch. |
| `schedule_delayed_action` | Stores a future action/effect on a slot, side, field, or creature. | Future Sight, Doom Desire, Wish, Healing Wish, Lunar Dance, delayed attacks/heals. |

### 3.3 Damage, Accuracy, Hit Count, And Criticals

| Primitive | What It Does | Covers |
|---|---|---|
| `accuracy_check` | Resolves hit/miss using move accuracy, stages, bypasses, weather, gravity, lock-on, and ruleset policy. | normal accuracy, sure-hit, No Guard-like effects, Lock-On, Gravity, weather accuracy changes. |
| `deal_damage` | Deals damage using a selected damage model. | standard, fixed, level, percent max HP, percent current HP, counter, stored, variable formula, OHKO. |
| `modify_base_power` | Changes base power before damage calculation. | Weather Ball, Gyro Ball, Low Kick, Facade, Acrobatics, terrain/weather boosts, item boosts. |
| `modify_damage` | Multiplies, reduces, blocks, redirects, or floors final damage. | STAB, type effectiveness, burn, screens, spread reduction, weather, resist berries, Life Orb, raid shields. |
| `modify_critical` | Changes crit chance, crit result, or crit behavior. | high-crit moves, Focus Energy, Laser Focus, Super Luck, crit stage policies. |
| `multi_hit_wrapper` | Repeats hit resolution multiple times with hit-count rules. | 2-5 hit moves, Triple Kick-like scaling, Population Bomb-like per-hit checks, Loaded Dice-like policies. |
| `set_hp_floor` | Prevents HP from dropping below a floor under conditions. | Endure, Sturdy, Focus Sash, affection survival, False Swipe-like cannot-KO damage. |

Damage model values for `deal_damage`:

| Model | Meaning |
|---|---|
| `standard` | Formula-based physical/special/status-derived damage. |
| `fixed` | Exact HP amount. |
| `level_based` | User level or level-derived formula. |
| `percent_max_hp` | Fraction of target max HP. |
| `percent_current_hp` | Fraction of target current HP. |
| `counter_received_damage` | Damage based on damage received earlier this turn. |
| `stored_damage` | Bide/Revenge-like released stored damage. |
| `variable_formula` | Formula from speed, weight, friendship, HP ratio, stat comparison, inventory, terrain, weather, target status, etc. |
| `ohko` | One-hit KO rules with level, immunity, accuracy, Sturdy-like blockers, and ruleset policy. |

### 3.4 HP Change, Costs, And Recovery

| Primitive | What It Does | Covers |
|---|---|---|

| `pay_cost` | Pays HP, PP, item, turn, fainting, stat, or condition cost. | Substitute, Belly Drum, Curse-Ghost, Mind Blown-like costs, Healing Wish self-faint. |
| `apply_recoil_or_crash` | Damages user after damage, miss, failure, or contact. | recoil, Jump Kick crash, Life Orb, Steel Beam-style cost. |
| `set_hp_by_formula` | Sets/equalizes HP using formula instead of normal damage/heal. | Pain Split, Endeavor-like HP matching. |
| `revive_target` | Restores fainted creature if rules allow. | Revive item, Revival Blessing-like effects. |

## 6. Condition Scopes

| Scope | Lifetime | Examples |
|---|---|---|
| `persistent_status` | Stored on creature instance; usually persists after switch/battle until cured. | burn, freeze, paralysis, poison, toxic, sleep. |
| `volatile` | Stored on active creature; usually clears on switch or battle end. | confusion, flinch, infatuation, partial trap, curse, leech seed, taunt, encore, substitute. |
| `side_condition` | Stored on one battle side. | hazards, Reflect, Light Screen, Safeguard, Mist, Tailwind, pledge effects. |
| `slot_condition` | Stored on active battle slot, not the creature. | Future Sight, Doom Desire, Wish, Healing Wish, Lunar Dance. |
| `field_condition` | Stored on shared battlefield. | weather, terrain, Trick Room, Magic Room, Wonder Room, Gravity, inverse rules. |
| `weather` | Specialized field condition with replacement/duration policy. | rain, sun, sandstorm, hail, snow, fog, strong winds, heavy rain, harsh sun. |
| `terrain` | Specialized field condition with grounded-only policy. | electric, grassy, misty, psychic. |
| `ruleset_condition` | Stored on battle format/ruleset state. | once-per-battle gimmick flags, raid shield state, obedience profile. |
| `form_state` | Stored on creature/form runtime state. | Mega-like, Gmax-like, Tera-like, Transform, Disguise break. |
| `item_state` | Stored on creature/item runtime state. | consumed berry, choice lock, item suppression, Knock Off state. |
| `ability_state` | Stored on creature/ability runtime state. | suppressed, copied, swapped, ignored, cannot-ignore. |
| `map_condition` | Stored on map/player state. | fog removed, boulder moved, cut tree removed, surf mode, repel steps. |

## 10. Coverage Checklist By Mechanic Family

A move/item/ability/field effect is supported only when every family it uses is implemented.

### 10.1 Action And Turn Control

- Priority and priority blockers.
- Turn-order reversal and order overrides.
- Target redirection, ally targeting, side targeting, random targeting, and spread targeting.
- Move charging, semi-invulnerability, delayed attacks, recharge, and skipped-charge policies.
- Move locking, forced repeats, same-move prevention, specific move disabling.
- Move availability replacement with Struggle-like fallback.
- Obedience and disobedience outcomes.
- Sleep/freeze/paralysis/confusion/flinch action prevention.
- First-turn-only, last-move-based, last-damage-based, and fail-if-repeated rules.
- Protection success decay and protection filters.
- Copy/call/repeat/exclusion move pools.

### 10.2 Damage Families

- Standard damage.
- Fixed damage.
- Level-based damage.
- Percent max HP damage.
- Current HP fraction damage.
- HP equalization/matching damage.
- Variable formula damage: weight, speed, friendship, HP ratio, stat ratio, inventory, terrain, weather, target status, target weight/height/metrics.
- Multi-hit damage and hit-count manipulation.
- Consecutive-use scaling.
- Counter/mirror damage.
- Stored/released damage.

## 11. True One-Offs And How To Handle Them

Some mechanics are very specific. They should not shape the core engine unless they reveal a reusable timing/scope model.

| Mechanic Type | Handling |
|---|---|
| Single-species story mechanics | Form/species condition or event flag. Do not change global resolver. |
| Single-ability/single-item exceptions | Ability/item hook with filters; no new primitive unless a new hook timing is required. |
| Boss/raid shields | Optional ruleset condition using `ruleset_check`, `modify_damage`, `set_hp_floor`, and scripted threshold hooks. |
| Totem/aura/stat intro boosts | Battle-start ruleset/side/creature condition. |
| Doubles-only attachment/merge mechanics | Optional `attach_ally` condition after doubles exists; disabled before then. |
| Legends-style action styles | Optional `action_style_modifier` ruleset profile; disabled by default. |
| Legends/Z-A trap reinterpretations | Preserve as alternate ruleset trap profile; do not replace normal hazard model. |
| Old contest effects | Out of battle scope unless contests become planned. Preserve metadata, do not execute. |
| TCG/anime/Mystery Dungeon/GO-only mechanics | Preserve as unsupported metadata unless explicitly planned. |
| Visual-only move effects | Presentation animation data, not battle effect primitives. |
| Hyper-specific official edge case | Preset + condition hook first; new primitive only after design review. |

### One-Off Decision Tree

```text
Can existing primitive + params express it?
  yes -> preset.
  no -> Can condition hooks express it?
    yes -> condition definition.
    no -> Can a formula/filter/tag express it?
      yes -> formula/filter/tag addition.
      no -> Does it introduce a new timing/scope model used by more than one mechanic?
        yes -> propose new primitive in BATTLE_SYSTEM_SPEC.
        no -> preserve unsupported metadata or implement as optional ruleset profile after review.
```

## 15. Source Notes

Research references used for coverage and normalization:

- PokeAPI move schema: move data exposes accuracy, effect chance, PP, priority, power, type, target, damage class, stat changes, ailment/category, hit counts, turn counts, drain/recoil, healing, critical rate, ailment chance, flinch chance, and stat chance.
- Bulbapedia move documentation: moves have battle and overworld uses, categories, type, target behavior, PP, and interaction flags.
- Bulbapedia status documentation: non-volatile status persists after switching/battle until cured; volatile status can stack and usually clears through switching or battle end; generation/ruleset differences matter.
- Bulbapedia entry hazard/trap documentation: hazards/traps are side conditions that trigger on switch-in, can be removed/swapped by specific effects, and include Spikes, Stealth Rock, Toxic Spikes, Sticky Web, and steel hazard variants.
- Bulbapedia weather documentation: weather changes damage, residual damage, accuracy, move charge behavior, abilities/items, overworld setup, and evolution conditions; Gen IX replaces Hail with Snow in the mainline profile.
- Bulbapedia terrain documentation: terrain affects grounded battlers and interacts with moves, abilities, and items.
- Bulbapedia Terastal documentation: Tera-like mechanics require type replacement/override plus special STAB/ruleset policy, not a damage-only modifier.
- Pokemon Showdown condition data: implementation-shaped reference for hook timing such as residual damage, action blockers, toxic counters, confusion, flinch, partial trapping, delayed attacks, locked moves, item locks, and Dynamax/form state.

## 16. Final Coverage Verdict

This v0.5 catalog accounts for official-style mechanics by **family**, not by move name. That is the correct level for this project.

The recommended long-term Core model is:

- about 50 canonical primitive operations;
- a condition registry with scoped hooks;
- a move/item/ability/form preset registry;
- a move tag/filter system;
- ruleset profiles for generation/custom differences;
- disabled preserved metadata for unsupported imports.

Do not expand the primitive list unless a mechanic requires a new timing/scope/resolver model. The primary risk is not missing one obscure move. The primary risk is letting obscure moves create one-off code paths.
